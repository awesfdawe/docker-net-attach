
## 1. Суть проекта (Elevator Pitch)
`docker-net-attach` — это легковесный, событийно-ориентированный (event-driven) сетевой контроллер для Docker. Он реализует паттерн **Zero Trust / Microsegmentation** для локальных инфраструктур (homelab / single-node deploy).

**Проблема:** В классическом сетапе с reverse-proxy (например, Traefik) все приложения помещаются в общую сеть (обычно `proxy_net`). Это создает критическую уязвимость: при компрометации одного контейнера злоумышленник получает горизонтальный доступ (lateral movement) ко всем остальным контейнерам в этой сети, включая бэкенды и базы данных.
**Решение:** Каждое приложение изолируется в собственной уникальной сети. `docker-net-attach` динамически, в рантайме, "прокидывает" шлюз (Traefik/Mihomo) строго в ту сеть приложения, которая явно указана администратором, основываясь на лейблах и жизненном цикле (health-статусе) контейнера.



## 2. Стек технологий
* **Python 3.14+** (Асинхронная архитектура).
* **httpx** (Для общения с Docker Engine API напрямую через Unix Domain Socket `/var/run/docker.sock`).
* **loguru** (Для структурированного логирования).
* **asyncio** (Event loop для неблокирующего чтения потока `/events`).

## 3. Как это работает (Core Logic)

Приложение функционирует по декларативной модели, требуя строгую пару лейблов для активации:
1. `docker-net-attach.<gateway_name>.enable=true` (Флаг активации для конкретного шлюза).
2. `docker-net-attach.<gateway_name>.network="название_сети"` (Явное указание целевой сети, можно несколько через запятую).

Контроллер работает в двух режимах одновременно:
1. **State Reconciler (Синхронизация состояния):** Выполняется при запуске. Запрашивает у Docker API текущее состояние всех контейнеров. Вычисляет разницу между «Фактическим» и «Желаемым» состоянием. Удаляет "остаточные" подключения шлюзов и создает недостающие, строго сверяясь с указанными целевыми сетями.
2. **Event Watcher (Реактор):** Подписывается на асинхронный стрим `docker events` и реагирует на изменения в реальном времени.

**Сценарий подключения (Attach):**
* Условие: Наличие обязательной пары лейблов и подтверждение того, что контейнер сам состоит в указанной сети.
* Событие: `start` (если нет healthcheck) или `health_status: healthy`.
* Действие: Контроллер выполняет `network connect`, привязывая шлюз к целевой сети.

**Сценарий отключения (Detach):**
* Событие: `die`, `stop` или `health_status: unhealthy`.
* Действие: Контроллер мгновенно отрезает целевую сеть приложения от шлюза (`network disconnect`), минимизируя вектор атаки при падении сервиса или компрометации.

## 4. Что учтено в архитектуре (Edge Cases & Security)

Для надежной работы в production-подобной среде реализована обработка следующих архитектурных нюансов:

* **Защита от мульти-сетевых утечек (Network Validation):** Если контейнер состоит в двух сетях (например, фронтенд и защищенная БД), контроллер подключит шлюз *только* к той сети, которая указана в `docker-net-attach.<gateway_name>.network`. Перед подключением выполняется жесткая валидация: скрипт проверяет, что запрошенная сеть действительно принадлежит данному контейнеру.
* **Self-Healing при перезапуске шлюзов:** Если контейнер Traefik пересоздается (например, при обновлении образа), Docker "забывает" все динамически подключенные к нему сети. Контроллер перехватывает событие `start` самого шлюза и запускает глобальный `sync_state()`, восстанавливая топологию.
* **Борьба со State Desync (Остаточные сети):** Если контроллер был выключен, а какие-то контейнеры удалились, при следующем запуске Reconciler найдет осиротевшие сети на стороне шлюза и принудительно их отключит (Force Disconnect).
* **ID vs Names:** Взаимодействие с API использует имена шлюзов (`"Container": "traefik"`), а не закешированные ID, предотвращая ошибки `404 Not Found` при пересоздании шлюзов.
* **Мульти-шлюзность (Multi-Gateway):** Через `ENV` передается список разрешенных шлюзов (`ALLOWED_GATEWAYS=traefik,mihomo`). Контейнер может иметь лейблы для подключения к нескольким прокси одновременно.
* **Dry Run:** Безопасный режим эмуляции. При включении логирует расчетные действия `Reconciler` и `Watcher`, не отправляя мутирующие POST-запросы в API Docker.
